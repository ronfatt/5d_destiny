# AI Destiny Platform Architecture V1

## 1. 文档定位

本文件定义五维命运系统的第三体系：`AI 占卜平台逻辑`。

目标是把前两套系统真正转成可以运行的产品架构：

- 第一体系：五维数学模型
- 第二体系：50 张命运卡系统

第三体系负责回答的问题不是“算什么”，而是：

- 用户怎么进入系统
- 数据怎么流动
- 哪些模块负责哪些计算
- AI 在哪里解释，哪里不能乱解释
- 报告如何被生成、展示和商业化

## 2. 平台目标

平台目标不是做一个单次算命页，而是做一个可持续运行的命运分析系统。

平台必须同时满足：

- 可输入
- 可计算
- 可抽牌
- 可解释
- 可生成报告
- 可复访
- 可付费

## 3. 六大核心模块

整个平台分为六个核心模块：

1. 用户输入系统
2. 命盘计算引擎
3. 卡牌随机引擎
4. 五维评分引擎
5. AI 解读引擎
6. 报告生成系统

## 4. 总流程

```text
用户输入
-> 命盘计算
-> 抽牌
-> 五维评分
-> AI 解读
-> 报告生成
-> 用户查看 / 保存 / 复访
```

## 5. 模块一：用户输入系统

### 5.1 目标

采集生成命盘、计算五维变量、触发 AI 解读所需的基础数据。

### 5.2 首次输入字段

基础字段：

- 出生日期
- 出生时间
- 出生城市
- 性别
- 当前问题

示例：

- 出生日期：`1998-05-22`
- 出生时间：`14:35`
- 出生城市：`Kuala Lumpur`
- 性别：`Female`
- 当前问题：`今年事业发展如何？`

### 5.3 平台层必要补充字段

为了让第三体系真正接上第一体系，平台不能只收出生信息，还必须支持以下输入：

- 当前主题：`career / wealth / love / health`
- `Mind` 问卷或 AI 心念访谈结果
- `Action` 问卷或 AI 行为访谈结果
- 可选：当前情绪标签
- 可选：最近 30 天行为标签

说明：

- 命盘只能提供 `Structure`
- 时运计算提供 `Timing`
- 卡牌提供 `Energy`
- 问卷 / 对话必须补齐 `Mind` 与 `Action`

### 5.4 输入模式

建议支持三种模式：

#### 标准模式

用户填写出生信息 + Mind/Action 问卷 + 当前问题。

#### 快速模式

用户只输入问题并抽牌，系统生成轻量版趋势报告。

#### 深度模式

用户输入完整出生信息、问卷、主题偏好，并生成多主题完整报告。

## 6. 模块二：命盘计算引擎

### 6.1 目标

根据出生信息生成紫微斗数命盘，并输出可进入五维模型的结构变量。

### 6.2 核心任务

1. 生成 12 宫位
2. 计算 14 主星
3. 计算辅星
4. 计算四化
5. 输出宫位原始分
6. 汇总四大主题结构分
7. 输出对应时运输入

### 6.3 输出结构示例

```json
{
  "palaces": {
    "life": 78,
    "career": 84,
    "wealth": 70,
    "relationship": 62,
    "fortune": 75,
    "travel": 68
  },
  "structure": {
    "career": 81,
    "wealth": 73,
    "love": 60,
    "health": 76
  },
  "timingBase": {
    "career": 1.12,
    "wealth": 0.98,
    "love": 1.03,
    "health": 0.95
  }
}
```

### 6.4 输出与第一体系的对应关系

- `structure.*` -> `Structure`
- `timingBase.*` -> `Timing`

## 7. 模块三：卡牌随机引擎

### 7.1 目标

从 `50` 张命运卡中抽取标准牌阵，并计算 `CardEnergyRaw` 与最终 `Energy`。

### 7.2 标准抽牌结构

每次占卜抽：

- `1` 张主星卡
- `2` 张能量卡
- `1` 张事件卡

示例：

```json
{
  "archetype": "武曲执行",
  "energy1": "机会",
  "energy2": "压力",
  "event": "合作"
}
```

### 7.3 引擎输出

```json
{
  "cards": {
    "archetype": { "id": "archetype.wuqu_execution", "value": 7 },
    "energy1": { "id": "energy.opportunity", "value": 5 },
    "energy2": { "id": "energy.pressure", "value": -3 },
    "event": { "id": "event.cooperation", "value": 4 }
  },
  "cardEnergyRaw": 3.4,
  "energy": 10.2
}
```

说明：

- `cardEnergyRaw` 是牌阵原始结果
- `energy` 是映射到第一体系 `Energy` 变量后的结果
- 映射规则以 [destiny-card-system-v1.md](/Users/rms/Desktop/Ai Project/Ai5Ddestiny/docs/destiny-card-system-v1.md) 为准

## 8. 模块四：五维评分引擎

### 8.1 目标

整合 `Structure / Timing / Energy / Mind / Action` 五个变量，输出各主题最终趋势分。

### 8.2 输入来源

- `Structure`：命盘引擎
- `Timing`：命盘时运模块
- `Energy`：卡牌引擎
- `Mind`：问卷或 AI 对话结果
- `Action`：问卷或 AI 对话结果

### 8.3 平台层必须区分两种公式

#### 展示公式

`Destiny Score = ((Structure × Timing) + Energy) × Mind + ActionDisplay`

适合：

- 课程讲解
- 用户教育
- 前端说明

#### 引擎公式

平台实际计算必须使用参数表中的稳定公式：

- `S = Structure / 100`
- `T = Timing`
- `E = 1 + (Energy / 100)`
- `M = Mind`
- `A = Action / 100`

中间指标：

- `BasePotential = S × T`
- `StateMomentum = E × M`
- `ExecutionPower = A × M`

最终：

`TrendScore = round((BasePotential × 50) + (StateMomentum × 20) + (ExecutionPower × 30))`

### 8.4 输出主题分

系统至少输出四个主题：

- `career`
- `wealth`
- `love`
- `health`

示例：

```json
{
  "scores": {
    "career": 104,
    "wealth": 82,
    "love": 58,
    "health": 76
  }
}
```

说明：

- 如果使用引擎分，分数应以参数表范围为准。
- 如果前端显示的是“展示分”，需要明确标记。
- 不建议前端混用两套分值而不说明。

## 9. 模块五：AI 解读引擎

### 9.1 目标

读取命盘、牌阵、问卷、分数结果，并生成：

- 趋势解释
- 主导变量
- 风险提示
- 行动建议

### 9.2 AI 输入结构

AI 必须读取结构化数据，不应直接根据自然语言自由发挥。

建议输入：

```json
{
  "question": "今年事业发展如何？",
  "theme": "career",
  "structure": 81,
  "timing": 1.12,
  "energy": 10.2,
  "mind": 1.06,
  "action": 72,
  "trendScore": 94,
  "cards": ["武曲执行", "机会", "压力", "合作"],
  "riskFlags": ["短期压力偏高"],
  "dominantFactor": "结构主导"
}
```

### 9.3 AI 输出目标

AI 输出应包含：

- 总结段
- 四维或单主题分析段
- 卡牌解释段
- 风险段
- 建议段

### 9.4 示例输出

事业分析：

当前事业结构较强，官禄宫与命宫配置较好，
本年度时运处于上升阶段。

抽到武曲执行卡与机会卡，
说明现实中存在新的商业入口。

压力卡提示短期负担较大，
但合作事件卡说明资源并非单打独斗获得，
更适合通过协作与执行推动突破。

综合趋势分较高，
当前属于可推进、可兑现、但不宜分散精力的阶段。

### 9.5 AI 约束规则

AI 不应：

- 虚构命盘细节
- 编造不存在的牌义
- 忽略结构化输入直接乱写
- 把轻微信号说成绝对命运

AI 必须：

- 优先解释已有分数与输入
- 明确区分结构、时运、能量、心念、行动的作用
- 给出可执行建议，不只给情绪安慰

## 10. 模块六：报告生成系统

### 10.1 目标

把结构化计算结果和 AI 解读包装成可交付产品。

### 10.2 报告结构

建议固定为：

1. 命盘概览
2. 五维指数
3. 卡牌解析
4. 四大主题分析
5. 风险提醒
6. 行动建议

### 10.3 输出形式

建议支持三种：

- 网页报告
- PDF 报告
- 日报 / 周报 / 月报摘要

### 10.4 报告层分级

#### 免费版

- 单主题摘要
- 基础分数
- 简短建议

#### 单次付费版

- 完整单主题报告
- 卡牌解析
- 风险提示
- 行动建议

#### 会员版

- 每日命运指数
- 每周能量更新
- 每月主题报告
- 年度命盘概览

## 11. 用户产品流程

### 11.1 基础流程

```text
Step 1: 打开平台
Step 2: 输入出生信息和问题
Step 3: 完成 Mind / Action 问卷
Step 4: 系统生成命盘
Step 5: 用户点击抽取命运卡
Step 6: 系统计算五维分数
Step 7: AI 生成完整报告
Step 8: 用户查看、保存或升级付费版本
```

### 11.2 快速模式流程

```text
Step 1: 输入问题
Step 2: 抽牌
Step 3: AI 输出轻量趋势
Step 4: 引导补填出生信息和问卷以升级完整版
```

## 15. 平台层最关键的工程原则

### 15.1 分层明确

- 命盘负责 `Structure/Timing`
- 卡牌负责 `Energy`
- 问卷负责 `Mind/Action`
- AI 只负责解释，不负责篡改分数

### 15.2 引擎先于文案

平台先跑结构化结果，再生成解读文案。

不允许先让 AI 随便写，再反推结果。

### 15.3 展示层与计算层分开

- 课程或产品文案可用简洁公式
- 平台内部必须用稳定引擎公式

### 15.4 所有结果可追溯

用户看到的每个结果，理论上都应能追溯到：

- 命盘哪部分导致
- 哪张牌导致
- 哪道问卷导致
- 哪个规则映射导致
